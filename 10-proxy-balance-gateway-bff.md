# Промежуточные слои обработки запросов

Прокси-серверы, балансировка, API Gateway и BFF

<!-- .slide: style="text-align: center" -->

Note:
В этой лекции разбираем архитектурный слой на входе в систему: прокси, балансировщики, API Gateway и BFF. Отдельный акцент делаем на ограничении трафика, потому что без него любая внешняя точка входа быстро становится уязвимой к перегрузке и abuse-сценариям.

Логика лекции такая: сначала сеть и трафик (proxy/LB/limiting), затем API-уровень (Gateway/BFF), после этого CQRS как инструмент разделения read/write и в конце сквозная сборка решений в один практический roadmap.

Термины: `abuse` — злоупотребление API (боты, скрейпинг, агрессивные запросы), `LB` — load balancer, то есть балансировщик нагрузки.

Типовой путь запроса:

`Client -> Edge/Reverse Proxy -> Gateway/BFF -> Services`

Мы сознательно не смешиваем входной API-слой с темами кэширования и хранения контента, чтобы сохранить фокус на управлении внешним трафиком и контрактом API.

Цепочка запроса понадобится дальше почти на каждом слайде. На ней удобно отмечать, где именно применяются health checks, rate limits, auth-политики и клиентская агрегация, чтобы роли слоёв не смешивались.

К концу лекции эту же цепочку можно использовать как чек-лист при проектировании новой системы: что обязательно на edge-слое, что переносить в gateway, а что оставлять в прикладных сервисах. Это даёт практический выход, а не только теорию.

Термины: `edge` — внешний периметр системы, куда сначала приходит клиентский трафик; `health checks` — проверки живости и готовности инстанса; `rate limit` — ограничение числа запросов за интервал.

## Прокси-серверы

<!-- .slide: style="text-align: center" -->

Note:
В первом блоке закладываем фундамент: что делает прокси-слой, чем отличается forward/reverse-модель и где проходит граница между транспортной инфраструктурой и API-паттернами.

Результат блока: после него должно быть понятно, где заканчивается задача сетевого периметра и почему не стоит пытаться «дотягивать» proxy до полноценного API-слоя.

---

## Что такое прокси и где он применяется

Прокси — промежуточный узел, который:

* принимает запрос;
* может фильтровать/модифицировать его;
* перенаправляет запрос дальше.

Практическое применение:

* TLS termination;
* централизованный логинг;
* базовые защитные и routing-политики.

Note:
Важно, что прокси не обязан быть «тупым ретранслятором». В продакшене он обычно становится точкой, где концентрируются периметральные политики: шифрование, фильтрация, заголовки, первичный контроль трафика.

При этом доменная бизнес-логика сюда не переносится. Как только в прокси начинает «жить» логика предметной области, слой становится сложным и превращается в архитектурный bottleneck.

Типовой пример из практики: ingress-proxy завершает TLS, добавляет технические заголовки трассировки и применяет базовые deny/policy-правила. После этого запрос должен попасть в сервис без «скрытой бизнес-магии» на edge-узле.

Термины: `TLS termination` — завершение шифрованного соединения на прокси; `routing` — выбор, куда отправить запрос дальше.

## Forward proxy vs Reverse proxy

* `Forward proxy` — действует от имени клиента и скрывает клиента от внешнего сервера.
* `Reverse proxy` — стоит перед сервисами и скрывает внутреннюю топологию от клиента.
* Для backend-систем чаще критичен именно reverse proxy.

Note:
Это ключевое терминологическое разделение. Forward proxy чаще встречается в корпоративном доступе к интернету, а reverse proxy — почти в каждой веб-платформе с внешним API.

На практике reverse proxy позволяет держать единый внешний адрес (`api.example.com`) при любом количестве внутренних сервисов. Это снижает связанность клиентов с внутренней архитектурой.

Если объяснять через аналогию, forward proxy помогает клиенту «выйти наружу» по корпоративным правилам, а reverse proxy помогает платформе «принять входящий трафик» по правилам сервера. Для backend-архитектуры нас почти всегда интересует второй сценарий.

## L4/L7 proxy и TLS-модели

`L4 proxy`:

* маршрутизация по IP/port;
* меньше контекста, выше простота.

`L7 proxy`:

* маршрутизация по host/path/headers;
* гибкие API-политики, canary и traffic shaping.

TLS-модели:

* `termination`;
* `passthrough`;
* `re-encryption`.

Note:
Компромисс простой: L4 быстрее и проще, L7 даёт больше управления API-потоком. Для современных API-платформ L7 обычно становится основой именно из-за политики и наблюдаемости.

По TLS нет универсального варианта. Termination удобен для централизованного управления сертификатами, а re-encryption нужен в инфраструктурах с жёсткими требованиями к шифрованию и во внутреннем сегменте.

Хороший практический критерий выбора такой: если достаточно равномерно раздать TCP-соединения, берём L4. Если нужны правила по URI/методу/заголовкам, постепенные релизы и тонкая маршрутизация, нужен L7.

Термины: `L4/L7` — уровни модели OSI (транспортный и прикладной), `canary` — выкладка на малую долю трафика, `traffic shaping` — управление распределением потока, `passthrough` — прокси не расшифровывает TLS, `re-encryption` — расшифровка и повторное шифрование.

## Forwarded headers и доверие

Ключевые заголовки:

* `X-Forwarded-For`;
* `X-Forwarded-Proto`;
* `X-Forwarded-Host`;
* `Forwarded`.

Правило доверия:

* доверять только заголовкам от доверенного proxy-слоя;
* не принимать клиентские forwarded-заголовки «как есть».

Граница ответственности:

* proxy решает транспортные и периметральные задачи;
* для единых API-политик и клиентской адаптации поверх него добавляют Gateway/BFF.

Note:
Без корректной обработки forwarded-заголовков backend начинает ошибаться в определении исходного IP и схемы, что ломает security-политику, audit и корректные redirect-ссылки.

Отдельный риск — spoofing: если слепо доверять клиентскому `X-Forwarded-For`, злоумышленник сможет подменить IP и обойти часть защит, включая примитивные rate limits.

Эта граница важна архитектурно: не каждый reverse proxy — это gateway. Часто система стартует с прокси+балансировки, а gateway/BFF добавляются позже, когда возникает потребность в более сложной политике API и клиент-специфичной агрегации.

Практический ориентир такой: если правило относится к transport/security perimeter (TLS, forwarded-заголовки, базовая маршрутизация), это зона proxy. Если правило относится к единому API-контракту, политикам доступа, клиентской агрегации и адаптации ответа, это уже зона Gateway/BFF.

Отдельно важно для цепочки из нескольких прокси: приложение должно учитывать только доверенные hop-узлы, иначе в поле с исходным IP можно получить недостоверное значение из внешнего трафика.

Термины: `spoofing` — подделка служебных данных запроса, `hop` — один промежуточный узел на пути запроса.

## Балансировка нагрузки

<!-- .slide: style="text-align: center" -->

Note:
Теперь переходим от одиночного входного узла к распределению трафика между несколькими инстансами. Цель — одновременно производительность, устойчивость и предсказуемость деградации.

Балансировка в этом блоке рассматривается как часть общей архитектурной устойчивости, а не как «магическое ускорение». Это важный акцент для корректных ожиданий команды.

---

## Балансировщик: роль и цели

Балансировщик:

* распределяет запросы между инстансами;
* исключает unhealthy-узлы из ротации;
* поддерживает масштабирование сервиса.

Цели:

* доступность;
* выравнивание утилизации;
* снижение tail latency (`p95/p99`).

Note:
Балансировка не просто «ускоряет». Её главная ценность — стабилизация поведения системы под нагрузкой: меньше горячих узлов, меньше резких провалов доступности.

Качество балансировки нужно оценивать по метрикам SLO: как ведут себя p95/p99 и error rate, а не только по среднему времени ответа.

Если упростить, балансировщик выполняет роль диспетчера: он не решает внутренние проблемы каждого сервиса, но помогает не допускать ситуаций, когда часть узлов перегружена, а часть простаивает.

Термины: `tail latency` — задержки в «хвосте» распределения, `p95/p99` — 95-й и 99-й перцентили времени ответа, `SLO` — целевой уровень сервиса (например, по latency и error rate).

---

## L4 vs L7 балансировка

`L4`:

* быстрый транспортный слой;
* уместен для простого routing и high-throughput.

`L7`:

* маршруты по API-контексту;
* canary/split traffic;
* пер-endpoint политики.

Note:
L4 хорошо подходит как базовый периметр, когда не требуется анализ HTTP-запроса. L7 нужен, когда помимо распределения важны API-правила, гибкая маршрутизация и управляемые релизы.

Во многих продакшен-системах эти уровни комбинируются: L4 для базового входа, L7 для API-логики.

Для объяснения студентам полезно подчеркнуть, что L7 обычно даёт больше контроля ценой большей сложности конфигурации и операционного сопровождения.

Термин: `high-throughput` — сценарий с очень большим потоком запросов/соединений в единицу времени.

---

## Алгоритмы балансировки: базовые

* `Round Robin` — равномерно по кругу, простой старт.
* `Least Connections` — новый запрос узлу с меньшим числом активных соединений.
* `Hash/Sticky` — локальность сессии ценой гибкости.
* `Weighted` — распределение по весам узлов.

Note:
У алгоритмов нет универсального победителя. Round Robin хорош в однородных кластерах, Least Connections — при разной длительности запросов, Weighted — в гетерогенной инфраструктуре.

Sticky-маршрутизация полезна для stateful-сценариев, но может ухудшить равномерность и отказоустойчивость. Поэтому её включают осознанно, а не по умолчанию.

Если в системе есть долгие соединения или «тяжёлые» запросы, Least Connections часто даёт более ровную картину нагрузки, чем классический Round Robin.

Термин: `sticky` (sticky-сессия) — закрепление клиента за одним и тем же инстансом на период сессии.

---

## Устойчивость балансировщика в эксплуатации

Что критично:

* `health checks` (`liveness`/`readiness`);
* защита от flapping;
* `connection draining` при деплое и scale-down.

Что балансировка сама не исправит:

* медленные бизнес-операции;
* узкие места БД;
* ошибочные retry/timeout-политики.

Note:
Даже хороший алгоритм не спасает, если readiness-check настроен плохо. Узел может быть «жив», но не готов к рабочему трафику.

Connection draining — обязательный operational baseline: сначала убрать узел из ротации, затем завершить текущие запросы, и только потом останавливать процесс. Иначе на каждом релизе будут всплески 5xx.

Балансировка перераспределяет трафик, но не ускоряет медленную логику сама по себе. Если SQL-запрос выполняется 2 секунды, он останется медленным на любом узле.

Это важный инженерный вывод: network-слой и application/data-слой нужно оптимизировать совместно.

Типичный пример: после добавления второго и третьего инстанса p50 улучшается, но p99 почти не меняется из-за медленного запроса к БД. В таком случае нужно оптимизировать запрос/индексы/пулы и retry-политику, а не ожидать, что балансировщик «вылечит» backend.

Чтобы уменьшить flapping, обычно добавляют пороговые значения и гистерезис: узел должен подтвердить стабильное состояние несколько проверок подряд, прежде чем вернуться в ротацию.

Термины: `liveness` — проверка, что процесс жив; `readiness` — проверка, что узел готов принимать трафик; `flapping` — частое переключение узла между healthy/unhealthy; `connection draining` — мягкое выведение узла из трафика без обрыва активных запросов.

## Ограничение трафика

<!-- .slide: style="text-align: center" -->

Note:
После балансировки переходим к управлению самим объёмом запросов. Limiting нужен, чтобы защитить систему от перегрузки, злоупотреблений и непредсказуемого роста стоимости.

На этом этапе важно донести, что limiting не только про безопасность, но и про управляемую деградацию: система должна замедляться предсказуемо, а не падать целиком.

---

## Зачем limiting и как различать термины

Зачем:

* защита от перегрузки и abuse;
* справедливое распределение ресурсов;
* контроль бюджета инфраструктуры.

Термины:

* `rate limit` — лимит за интервал;
* `throttling` — принудительное замедление;
* `quota` — лимит за длительный период;
* `concurrency limit` — лимит параллельных операций.

Note:
Без limiting один клиент способен «съесть» ресурсы и ухудшить SLA для остальных. Особенно это заметно в публичных API и B2B-интеграциях.

Важно не смешивать виды ограничений: можно укладываться в rate limit, но перегрузить систему избыточной параллельностью.

Полезный прикладной пример: клиент может отправлять только 5 запросов в секунду, но держать 100 долгих соединений. По rate limit это «норма», а по concurrency — реальная перегрузка.

---

## Что ограничивать и где ставить лимиты

По каким ключам:

* IP;
* user/account;
* `client_id`/token;
* tenant;
* endpoint + method.

На каких слоях:

* edge/reverse proxy;
* API Gateway;
* BFF;
* критичные backend-операции.

Note:
Один ключ и один слой почти всегда недостаточны. Для B2B-сценариев чаще важен `client_id` или tenant, для публичного API — добавляется per-IP защита.

Многоуровневый limiting даёт защиту в глубину: даже если один слой деградировал, контроль трафика остаётся на других уровнях.

Практически это означает политику вида: coarse-limit на edge, более точный бизнес-limit на gateway/BFF и локальная защита особенно дорогих backend-операций.

Термин: `tenant` — отдельный клиент/организация в multi-tenant системе, чьи лимиты и изоляция считаются отдельно.

## Hard/Soft limits и профиль нагрузки

* `Hard limit` — сразу `429`.
* `Soft limit` — замедление/очередь/приоритизация.

Нагрузка:

* `burst` — короткий всплеск;
* `sustained` — длительная повышенная нагрузка.

Note:
Hard limit проще и надёжнее защищает backend, но может резко ухудшать UX. Soft-подход мягче для пользователя, но сложнее в реализации и управлении задержками.

Политика должна учитывать и burst, и sustained-профили. Например, разрешить burst 20 req/s, но удерживать долгий поток на уровне 5 req/s.

В учебном примере можно сравнить два сценария: при hard-limit клиент сразу получает `429`, при soft-limit часть запросов проходит с увеличенной задержкой, но критические операции не блокируются полностью.

---

## Алгоритмы limiting: сравнение

* `Fixed Window` — просто, но есть boundary-burst.
* `Sliding Window` — ровнее и справедливее, но дороже.
* `Token Bucket` — контролируемые всплески.
* `Leaky Bucket` — выравнивание потока.

Note:
Fixed Window часто выбирают как старт, но на зрелой нагрузке его артефакты заметны: всплески на границах окна.

Token Bucket практичен для API с короткими пиками, Leaky Bucket полезен, когда downstream-сервисы чувствительны к резким рывкам потока.

Если нужна простота и быстрый запуск, начинают с Fixed/Token Bucket. Если важнее равномерность и защита чувствительных downstream-систем, переходят к Sliding/Leaky-подходам.

Термин: `boundary-burst` — всплеск запросов на границе двух окон подсчёта лимита.

---

## Распределённые лимиты и контракт ответа

Распределённая реализация:

* централизованные счётчики (например, Redis);
* атомарные операции (Lua/transactions).

Ответ API при превышении:

* `429 Too Many Requests`;
* `Retry-After`;
* `X-RateLimit-*` заголовки.

Note:
В распределённой системе локальные счётчики на узлах дают рассинхрон и обход лимитов. Централизованный atomic-подход снижает эту проблему.

Клиент должен получить понятный контракт, иначе начнутся агрессивные retry и self-inflicted overload.

Отдельный момент — согласованность времени и атомарность операций. Без них при высокой конкуренции легко получить двойной расход лимита или некорректный `remaining`.

Термины: `Retry-After` — заголовок, когда можно повторить запрос после `429`; `X-RateLimit-*` — служебные заголовки с текущими лимитами; `atomic` — операция выполняется неделимо и без промежуточных состояний.

## Observability и анти-паттерны limiting

Ключевые метрики:

* доля `429` по сегментам;
* доля ложноположительных блокировок;
* latency лимитирующего слоя;
* влияние на p95/p99 backend.

Анти-паттерны:

* одинаковый лимит для всех;
* отсутствие backoff у клиентов;
* отсутствие поэтапного rollout.

Note:
Limiting нельзя внедрять «вслепую». Нужен эволюционный режим: observe -> dry-run -> partial enforce -> full enforce.

Критически важно отслеживать не только защиту системы, но и ущерб для легитимного трафика.

На практике полезно начинать с ограничения на малой доле трафика и отдельными дашбордами по сегментам клиентов, чтобы быстро увидеть, кого именно режет новая политика.

Термины: `dry-run` — режим проверки без реальных блокировок, `backoff` — стратегия увеличения паузы перед повторными попытками, `rollout` — поэтапный ввод изменения.

## API Gateway

<!-- .slide: style="text-align: center" -->

Note:
Теперь поднимаемся на API-уровень: от транспортного контроля к управлению внешним контрактом и едиными политиками доступа.

Ключевая идея блока: после того как сеть и трафик стабилизированы, следующая проблема почти всегда про единообразие поведения API для разных клиентов и команд.

---

## API Gateway: определение и границы

API Gateway:

* единая внешняя точка входа;
* централизация API-политик;
* маршрутизация и унификация контракта.

`Gateway != proxy/lb`:

* proxy/lb — транспорт;
* gateway — API-политика и контрактный слой.

Note:
Физически gateway может использовать proxy-движок, но архитектурно это более высокий уровень ответственности.

Если пытаться решить API-политику только средствами «голого» proxy/lb, быстро появляется технический долг и разъезд поведения API.

Удобно формулировать так: proxy отвечает за доставку и периметр, gateway отвечает за правила «как внешний API должен себя вести» независимо от внутренней структуры сервисов.

---

## Типовой pipeline Gateway

1. authN/authZ;
2. rate limiting;
3. validation/policy;
4. routing;
5. response transform;
6. logging/tracing.

Note:
Стабильный pipeline делает API-поведение предсказуемым для всех команд и клиентов. Это особенно важно в крупных системах с большим количеством маршрутов.

При этом pipeline должен оставаться техническим. Доменную логику сюда переносить нельзя.

Хорошая практика — фиксировать pipeline как платформенный стандарт, чтобы новый маршрут автоматически наследовал базовые требования к безопасности, ограничениям и наблюдаемости.

Термины: `authN` — аутентификация (кто пользователь), `authZ` — авторизация (что ему разрешено), `tracing` — трассировка запроса через цепочку сервисов, `response transform` — техническое преобразование ответа без бизнес-логики.

---

## Что уместно и неуместно в Gateway

Уместно:

* кросс-срезовые API-политики;
* валидация и безопасность;
* общая observability.

Неуместно:

* доменные инварианты;
* stateful workflow;
* тяжёлая оркестрация бизнес-операций.

Note:
Главный критерий: если правило не зависит от доменной модели и повторяется на множестве маршрутов, это кандидат для gateway.

Если доменная логика растёт в gateway, он превращается в bottleneck и «fat gateway».

Проверочный вопрос для команды: если убрать конкретный бизнес-кейс, останется ли это правило полезным для большинства API? Если да, это gateway-уровень; если нет, правило лучше держать в сервисе.

Термин: `кросс-срезовые политики` — технические правила, одинаковые для многих API-маршрутов (безопасность, валидация, лимиты, логирование).

---

## Fat gateway и модели внедрения

Симптомы fat gateway:

* рост доменной логики;
* каскадные изменения;
* снижение скорости релизов.

Модели внедрения:

* single shared;
* federated;
* hybrid (edge + domain gateway).

Note:
Single-модель проще на старте, federated лучше масштабируется по командам, но требует зрелой платформенной дисциплины.

Выбор модели зависит не только от нагрузки, но и от организационной структуры ownership.

На ранних этапах часто выбирают single shared gateway ради скорости запуска. По мере роста числа команд переходят к hybrid/federated-модели, чтобы не блокировать релизы через одну центральную команду.

Термины: `federated` — распределённая модель ownership между командами, `hybrid` — сочетание общего edge-шлюза и доменных gateway-слоёв.

---

## Backend for Frontend (BFF)

<!-- .slide: style="text-align: center" -->

Note:
Дальше переходим к клиент-ориентированной адаптации API. BFF нужен, когда один общий контракт перестаёт одинаково хорошо обслуживать web, mobile и partner-сценарии.

Этот блок полезно рассматривать как продолжение Gateway: Gateway задаёт общие правила входа, а BFF адаптирует выдачу под конкретный клиентский контекст.

---

## BFF: что это и когда нужен

BFF:

* backend под конкретный тип клиента;
* оптимизация payload и формата под UX-сценарий.

Обычно нужен, когда:

* сильно расходятся сценарии web/mobile;
* разные требования к latency и объёму данных.

Note:
Пример: Web BFF может отдавать полный карточный набор, Mobile BFF — компактный payload без тяжёлых вторичных блоков.

Если клиентские потребности почти одинаковы, отдельный BFF-слой может быть избыточным.

То есть BFF вводится не «по шаблону», а при подтверждённой пользе: меньше round-trip, меньше лишних полей и более стабильный UX на каждом типе клиента.

Термины: `payload` — тело ответа API, `round-trip` — один полный цикл клиентского запроса и ответа.

---

## Gateway + BFF и клиентская агрегация

Частая схема:

`Edge Gateway -> BFF -> Internal Services`

Роли:

* gateway — общие политики;
* BFF — агрегация и адаптация контракта.

Note:
Такое разделение помогает избежать и дублирования политик, и перегруза общего gateway клиентской спецификой.

На практике BFF часто уменьшает число round-trip от клиента: вместо 3–4 вызовов клиент получает один агрегированный ответ.

Важно подчеркнуть границу: безопасность и общие ограничения централизуются в gateway, а формат и композиция ответа для конкретного интерфейса живут в BFF.

---

## Ownership BFF и анти-паттерны

Практика ownership:

* BFF владеет команда клиента.

Плюсы:

* быстрый цикл изменений;
* меньше межкомандных блокировок.

Риски:

* дублирование доменной логики в разных BFF;
* расхождение поведения клиентов.

Note:
Ownership ускоряет продуктовую разработку, но требует платформенных стандартов по безопасности, observability и versioning.

Профилактика дублирования — общие SDK и строгое разделение: доменная логика в сервисах, клиентская адаптация в BFF.

Если таких стандартов нет, несколько BFF быстро начинают расходиться по форматам ошибок, правилам ретраев и security-поведению, что повышает стоимость поддержки.

---

## CQRS на API-слое

<!-- .slide: style="text-align: center" -->

Note:
Теперь связываем входной слой с разделением read/write. CQRS здесь рассматривается как способ уменьшить конфликт между тяжёлыми read-сценариями и строгими write-инвариантами.

Задача блока — показать CQRS не как абстракцию, а как прикладной приём на API-слое, когда один и тот же endpoint-модуль одновременно должен быть «и быстрым, и строго консистентным».

Термин: `CQRS` (Command Query Responsibility Segregation) — разделение моделей и потоков чтения и записи.

---

## Почему CQRS и что разделяется

Проблема:

* в одном API смешаны сложные read-запросы и write-команды.

CQRS-разделение:

* `Command`-поток для изменений;
* `Query`-поток для чтения оптимизированных представлений.

Note:
CQRS не равен «две базы любой ценой». Главная идея — разделить ответственность моделей и сделать каждую сторону оптимальной под свою задачу.

На практике это часто означает отдельные маршруты и DTO для чтения и отдельные команды для записи, чтобы не усложнять write-инварианты ради удобства выдачи списка.

Термины: `Command` — операция изменения состояния, `Query` — операция чтения без изменения состояния.

---

## Read/Write-модели и консистентность

`Read-model`:

* быстрые выборки;
* DTO под клиентские сценарии.

`Write-model`:

* инварианты;
* транзакционные гарантии;
* идемпотентность команд.

Консистентность:

* write обычно строже;
* read может быть eventual.

Note:
Клиенту нужно явно объяснять поведение: после команды read-данные могут обновиться не мгновенно. Иначе появляются ложные баг-репорты и хаотичные retry.

Если система допускает eventual consistency, это должно быть отражено в API-контракте и документации: где ожидать задержку обновления, а где результат должен быть виден сразу.

Термины: `DTO` — объект передачи данных без доменной логики, `инвариант` — правило, которое всегда должно выполняться после операции записи, `идемпотентность` — повтор команды не меняет результат сверх первого успешного выполнения, `eventual consistency` — согласованность с задержкой.

---

## Когда CQRS оправдан

Оправдан, если есть:

* тяжёлые read-сценарии;
* явная read/write-асимметрия;
* организационная готовность поддерживать раздельные модели.

Не оправдан, если:

* система мала;
* нет подтверждённой боли;
* внедрение идёт «по моде».

Note:
CQRS добавляет сложность. Его стоит внедрять там, где выгода измерима: скорость read-path, устойчивость write-инвариантов, снижение operational-боли.

Если таких измеримых целей нет, разумнее оставить более простую модель и вернуться к CQRS позже, когда появятся данные о реальных ограничениях текущей архитектуры.

## Сборка в архитектуру

<!-- .slide: style="text-align: center" -->

Note:
Финальный блок собирает всю лекцию в прикладной формат: как из отдельных паттернов сложить рабочую архитектуру и как внедрять её без big bang-переписывания.

Фокус этого блока — не перечислить паттерны повторно, а связать их в последовательность внедрения, которую можно применить в реальном проекте.

## Сквозной пример + чек-лист выбора

Схема:

`Client -> Edge Proxy/LB -> API Gateway -> BFF -> Services`

Дополнительно:

* limiting на edge/gateway;
* CQRS для сложных read/write API-сценариев.

Чек-лист:

1. Где bottleneck?
2. Нужна ли единая API-политика?
3. Нужна ли клиентская адаптация API?
4. Нужен ли limiting на нескольких уровнях?
5. Есть ли основания для CQRS?

Note:
Сквозная схема нужна как карта ответственности: у каждого слоя свой класс задач и свой набор метрик.

Чек-лист помогает принимать архитектурные решения воспроизводимо, а не на уровне «интуитивных предпочтений».

Полезный формат обсуждения с командой: пройти чек-лист по одному конкретному проблемному endpoint и зафиксировать, какой слой должен решать каждую наблюдаемую боль.

---

## Roadmap внедрения без big bang

1. reverse proxy + health checks;
2. корректная балансировка + draining;
3. limiting в observe/dry-run;
4. API Gateway для единых политик;
5. BFF для разных клиентских каналов;
6. точечный CQRS там, где подтверждена боль.

Note:
Эволюционный подход уменьшает риск больших регрессий. Каждый шаг должен иметь свою метрику успеха: latency, error rate, 429 ratio, скорость релизов, число инцидентов.

Отдельно стоит зафиксировать критерии остановки на каждом этапе: если метрики не улучшаются, переход к следующему шагу roadmap откладывается до устранения причин.

---

## Итоги

* Proxy/LB — фундамент входного слоя.
* Limiting — защита стабильности и бюджета.
* API Gateway — единые политики и контроль контракта.
* BFF — клиент-специфичная адаптация API.
* CQRS — инструмент для сложных read/write-сценариев.

Note:
Главный вывод лекции: это не конкурирующие технологии, а совместимые слои ответственности. Их нужно подключать по мере роста требований, а не внедрять «полным комплектом» в первый день.

Это помогает избежать двух крайностей: недостроенного входного слоя на раннем этапе и избыточно сложной архитектуры без подтверждённой необходимости.
