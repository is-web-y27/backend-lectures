# RESTful API

Не CRUD-ом единым

<!-- .slide: style="text-align: center" -->

Note:
В этой лекции мы разбираем RESTful API как архитектурный подход, а не как набор случайных правил по URI и HTTP-методам. Мы пройдём путь от предыстории до практических решений: как выбирать методы, коды ответа, схему версионирования и документацию. 3. Ограничения REST и их практический смысл.

## От предыстории к REST

<!-- .slide: style="text-align: center" -->

Note:
Сейчас начинаем первый тематический блок. В нём разберём исторический контекст: что пытались делать до REST, какие ограничения были у прежних подходов и как появилась идея зрелой ресурсно-ориентированной модели API.

---

## До REST: что пытались делать

До REST многие подходы пытались подогнать HTTP под бизнес-операции.

Один из известных примеров: **WebDAV (Web Distributed Authoring and Versioning)**.

Идея:

* Добавить новые методы;
* Управлять удалёнными ресурсами как файлами/коллекциями.

Note:
Исторически инженерная интуиция была довольно прямой: если стандартных HTTP-методов не хватает, добавим новые, которые точнее отражают прикладные операции. WebDAV был как раз таким расширением, ориентированным на удалённую работу с документами и метаданными. Подход логичный и для своей предметной области полезный, но именно здесь хорошо видны первые компромиссы: чем больше вы отклоняетесь от стандартного HTTP, тем больше зависимость от инфраструктуры и поддержки в экосистеме.

---

## WebDAV: примеры методов

Дополнительные методы WebDAV:

* `COPY`, `MOVE`;
* `LOCK`, `UNLOCK`;
* `MKCOL`;
* `PROPFIND`, `PROPPATCH`.

Плюс: богатые операции из коробки для систем, работающих с файлами и/или документами.

Note:
Эти методы показывают, насколько WebDAV был ориентирован на файловую модель: коллекции, блокировки, свойства ресурсов, операции копирования и перемещения. В корпоративных системах совместной работы это действительно удобно и достаточно естественно. Но если смотреть на API широкого назначения, такой набор методов начинает плохо масштабироваться как универсальный подход. Для команд разработки это часто означает сложную интеграцию, дополнительные требования к прокси и нестандартное поведение клиентов.

---

## Ограничения WebDAV-подхода

Минусы:

* Нужна поддержка на стороне сервера/прокси;
* Экосистема клиентов и инструментов неоднородна;
* Сложнее межплатформенная совместимость;
* В общем случае получается частный протокол на базе HTTP.

Отсюда и переход к более универсальному подходу — REST.

Note:
Ключевой вывод из опыта WebDAV: расширять HTTP под каждую предметную область можно, но цена такой гибкости быстро растёт. Как только у вас появляются разные клиенты, мобильные приложения, сторонние интеграторы и сложная инфраструктура, нестандартные методы и нестандартная семантика становятся источником проблем. REST предложил более универсальный путь: не добавлять новые методы под каждую задачу, а использовать ограниченный набор стандартных возможностей HTTP вместе с чёткой ресурсной моделью.

---

## Уровни зрелости API

Модель зрелости Ричардсона:

* Level 0: один эндпоинт, один метод;
* Level 1: несколько URI, но по-прежнему один метод;
* Level 2: ресурсы + HTTP-методы + коды статусов;
* Level 3: гипермедиа как движок состояния приложения (**HATEOAS**).

Note:
Richardson Maturity Model полезна как диагностический инструмент: она не говорит, что ваш API "плохой" или "хороший", а помогает понять, на каком уровне зрелости вы находитесь. Для большинства производственных систем базовой целевой точкой является Level 2, потому что он уже даёт предсказуемый контракт и хорошо поддерживается всей экосистемой. Level 3 с гипермедией концептуально очень интересен, но на практике применяется значительно реже и не всегда окупает сложность.

---

## Level 0: RPC over HTTP

Характеристики:

* Один URI для всех операций;
* Часто только `POST`;
* Тип операции зашит в тело запроса.

Пример:

```http []
POST /api HTTP/1.1
Host: api.example.com
Content-Type: application/json
Accept: application/json

{ "action": "createUser", "payload": { ... } }
```

Note:
Level 0 — это по сути удалённый вызов процедур, просто доставленный через HTTP-транспорт. Формально такой API может работать быстро и решать задачу, но он почти не использует преимущества веба: промежуточные кэши, семантику методов, понятные коды ответа и стандартизированную наблюдаемость. Для клиентов это обычно означает плотную связку с внутренними деталями сервера и слабую самоочевидность контракта.

---

## Level 1: появились ресурсы

Характеристики:

* URI разделены по сущностям;
* Но методы используются однообразно (например, только `POST`).

Примеры:

```http []

Host: api.example.com
Content-Type: application/json

{ "id": "u-42" }
```

```http []
POST /users/create HTTP/1.1
Host: api.example.com
Content-Type: application/json

{ "name": "Alice", "email": "alice@example.com" }
```

</div></div>

Note:
На этом уровне уже появляется шаг вперёд: API начинает отражать предметную область через ресурсы. Но семантика операций всё ещё «спрятана» в URI и теле запроса, а не выражена через стандартные HTTP-методы. В результате проблемы Level 0 уменьшаются, но не исчезают: кэширование и повторные запросы обрабатываются хуже, а интеграции становятся менее предсказуемыми, чем могли бы быть в ресурсно-методной модели.

---

## Level 2: основа RESTful API

Характеристики:

* Ресурсы описываются URI;
* Семантика операций выражается HTTP-методами;
* Используются корректные HTTP status codes.

Именно этот уровень обычно подразумевают под "REST API" на практике.

Примеры:

<div style="display: grid; grid-template-columns: auto auto"><div>

```http []
GET /users/u-42 HTTP/1.1
Host: api.example.com
Accept: application/json
```

</div><div>

```http []
PATCH /users/u-42 HTTP/1.1
Host: api.example.com
Content-Type: application/json
If-Match: "v3"

{ "name": "Alice Cooper" }
```

</div></div>

Note:
Level 2 — это тот практический минимум, который уже даёт максимальную отдачу в реальных системах. Клиенты получают предсказуемый контракт, инфраструктура лучше понимает трафик, а команда быстрее согласовывает поведение API. Здесь становятся особенно важны дисциплина именования, корректный выбор методов и аккуратная работа со статус-кодами. Когда эти вещи сделаны последовательно, API становится устойчивым к росту команды и количества интеграций.

---

## Level 3: HATEOAS

Идея:

* Ответ содержит не только данные, но и ссылки/действия;
* Клиент "навигационно" двигается по API.

Плюс:

* Сильная самодокументируемость интерфейса.

Минусы:

* Реализуется редко;
* Мало где окупается для machine-to-machine клиентов.

Note:
HATEOAS означает, что сервер в ответе подсказывает клиенту допустимые следующие шаги. В теории это снижает связанность и позволяет менять внутреннюю структуру API без жёсткого переписывания клиентов. На практике большинство интеграций — это заранее запрограммированные клиенты, которым проще работать по стабильной схеме URI и контрактов, поэтому полноформатный Level 3 встречается нечасто. Но сам подход полезен как идея: ответы API должны быть максимально самообъяснимыми.

    "self": { "href": "/users/u-42" },
    "orders": { "href": "/users/u-42/orders" },
    "deactivate": { "href": "/users/u-42/deactivation", "method": "POST" }
  }

Даже если в будущем часть маршрутов изменится, сервер может сохранить устойчивость клиентского поведения через корректные `_links`. В реальной жизни такой подход чаще используют частично: например, для ключевых переходов в сложных workflow.

## Архитектурные основы REST

<!-- .slide: style="text-align: center" -->

Note:
Переходим ко второму блоку. Здесь фиксируем фундамент: что такое REST как архитектурный стиль, какие у него ограничения и почему именно они делают API предсказуемым, масштабируемым и удобным для интеграций.

---

## Что такое REST и RESTful

**REST (Representational State Transfer)** — архитектурный стиль.

Термин и подход сформулировал Рой Филдинг в 2000 году в своей докторской диссертации об архитектуре Web.

Важно:

* Это не протокол;
* Можно применять не только поверх HTTP;
* **RESTful API** — API, которое следует REST-ограничениям.

Note:
Частая ошибка — воспринимать REST как «спецификацию эндпоинтов». Исторически REST появился в 2000 году в диссертации Роя Филдинга как попытка формализовать архитектурные принципы, на которых уже работала Всемирная паутина. На самом деле REST — это архитектурные ограничения, из которых вытекают практические решения по интерфейсу. HTTP просто оказался идеальной средой для применения этих ограничений, потому что уже содержит методы, коды, заголовки и ресурсо-ориентированную модель. Поэтому формулировка «RESTful» означает не наличие слова REST в документации, а соответствие ключевым ограничениям стиля.

---

## Ограничения REST

Ключевые ограничения:

* Клиент-серверная архитектура (Client-Server, web/mobile-клиенты работают с единым API-контрактом);
* Отсутствие состояния (Stateless, каждый запрос несёт всё нужное для обработки, например, авторизацию);
* Кэшируемость (Cacheable, ответы могут отдавать заголовки кэширования);
* Единообразный интерфейс (Uniform Interface, единая семантика запросов);
* Поддержка нескольких звеньев (Layered System, клиент может ходить к серверу не напрямую);
* Предоставление исполняемого кода (Code-on-Demand, опционально).

Note:
Смысл ограничений в том, что они снижают сложность системы на масштабе. Важно, что это не абстракция: каждое ограничение напрямую влияет на проектные решения и эксплуатацию API. `Stateless` и `Layered System` помогают масштабировать систему и упростить инфраструктуру, `Uniform Interface` делает контракт предсказуемым, а `Cacheable` даёт выигрыш по задержкам и нагрузке. Если команда осознанно применяет все ограничения вместе, API получается не только «красивым», но и устойчивым в продакшене.

---

## Единообразный интерфейс

Компоненты:

* Идентификация ресурсов (URI);
* Работа через представления ресурса (JSON/XML/...);
* Самоописываемые сообщения (метод, заголовки, media type, status code);
* Гипермедиа как механизм переходов (HATEOAS).

Note:
Uniform Interface — центральная идея REST, потому что именно она делает API общепонятным для любых клиентов и инструментов. Клиент должен понимать, что он запрашивает, в каком виде пришёл ответ и какие действия допустимы дальше, не опираясь на скрытые соглашения. Чем лучше выдержан единообразный интерфейс, тем дешевле поддержка API, проще онбординг разработчиков и ниже риск неявных регрессий при изменениях.

---

## Ресурс в REST

Ресурс:

* Сущность или набор сущностей, доступных по URI;
* Не "метод", а объект работы API.

Связь с DDD:

* Ресурс часто отображает агрегат или сущность;
* Не обязан быть 1:1 с таблицей БД.

Note:
Ресурсная модель нужна, чтобы API описывал предметную область, а не внутренние технические шаги сервера. Хороший ресурс отражает бизнес-смысл: пользователь, заказ, платёж, инвойс, а не «runProcess» или «executeAction». В контексте DDD ресурс часто соответствует агрегату или сущности, но не стоит механически привязывать его к структуре таблиц: контракт API проектируется для потребителей, а не как зеркальное отражение хранилища.

## Проектирование API-контрактов

<!-- .slide: style="text-align: center" -->

Note:
Это самый прикладной блок. Здесь собраны практические решения, которые напрямую влияют на качество API-контракта: именование, методы, статусы, формат ошибок, content negotiation, фильтрация, пагинация и другие детали, от которых зависит удобство интеграции.

---

## Именование URI

Хорошие практики:

* Имена существительные, не глаголы;
* Во множественном числе для коллекций;
* Дефисы для составных слов;
* Без расширений файлов.

Примеры:

* `GET /users`
* `GET /users/{id}`
* `GET /order-items`

Note:
Нейминг URI — это не косметика, а один из главных факторов предсказуемости API. Когда правила именования едины, интеграторы быстрее понимают структуру сервиса и делают меньше ошибок. Глаголы обычно лучше выражать методами HTTP, а URI оставлять ресурсными. Консервативный и единый стиль именования почти всегда выигрывает у «креативных» исключений, даже если локально они кажутся удобными.

---

## Вложенные ресурсы

Вложенные URI полезны, когда дочерний объект имеет явный контекст родителя:

* `GET /users/{userId}/orders`
* `GET /orders/{orderId}/items`

Практические правила:

* Избегать глубокой вложенности (обычно не глубже 2 уровней);
* Если у ресурса есть собственный жизненный цикл, дать ему «плоский» root-endpoint;
* Для фильтрации и выборок чаще использовать query-параметры, а не лишние уровни URI.

Note:
Вложенные ресурсы легко делают API «читаемым», но также легко превращают его в длинные и хрупкие маршруты. Хороший ориентир: если сущность действительно существует только в контексте родителя, вложенность уместна. Если же сущность самостоятельна и на неё есть независимые сценарии, лучше выделить отдельный корневой ресурс и связывать объекты через идентификаторы и фильтры. Это помогает избежать разрастания маршрутов и упрощает поддержку клиента.

---

## Операции над ресурсами

Базовый набор:

* `GET /users` — список;
* `GET /users/{id}` — элемент;
* `POST /users` — создание;
* `PUT /users/{id}` — полная замена;
* `PATCH /users/{id}` — частичное обновление;
* `DELETE /users/{id}` — удаление.

Note:
Базовый CRUD-набор важен не потому, что это «традиция», а потому что он хорошо согласуется с семантикой HTTP и ожиданиями клиентов. Особенно полезно явно проговаривать разницу между `PUT` и `PATCH`: первый про полное представление ресурса, второй про частичное изменение. Когда эта грань размыта, начинаются ошибки с идемпотентностью, валидацией и повторной доставкой запросов.

---

## Массовые операции над ресурсами

Когда нужны массовые операции:

* Массовые изменения одной сущности;
* Снижение числа round-trips;
* Асинхронная обработка больших пачек.

Практика:

* `POST /user-deactivation-jobs` — создать задачу массовой деактивации;
* `POST /user-deletion-jobs` — создать задачу массового удаления;
* `GET /user-deletion-jobs/{jobId}` — проверить статус выполнения;
* Для длительных задач: `202 Accepted` + `Location` на URI задачи.

Note:
Bulk-операции не противоречат REST, если оформлены как работа с отдельным ресурсом-задачей, а не как RPC-команда в URI. Хороший шаблон: принимать список идентификаторов и параметры операции, валидировать каждую запись и возвращать агрегированный результат. Важно заранее определить поведение при частичных ошибках: либо «all-or-nothing», либо частичный успех с отчётом по каждому элементу. Для тяжёлых задач лучше сразу уходить в асинхронный режим через job-resource.

---

## Оставшиеся HTTP-методы

Кроме CRUD-методов, важно понимать:

* `HEAD` — как `GET`, но без тела ответа;
* `OPTIONS` — какие методы/настройки доступны ресурсу;
* `TRACE` — диагностический echo-запрос (обычно отключают в проде);
* `CONNECT` — создание туннеля (обычно прокси-сценарий, не бизнес-API).

Практически для REST API чаще всего применяются `HEAD` и `OPTIONS`.

Note:
Этот слайд нужен, чтобы картина HTTP была полной. В бизнес-API мы действительно чаще используем набор `GET/POST/PUT/PATCH/DELETE`, но это не значит, что остальные методы не существуют. `HEAD` полезен для метаданных и проверки изменений без передачи тела, `OPTIONS` полезен для discovery и CORS, а `TRACE` и `CONNECT` обычно находятся за пределами типичного REST-контракта приложения. Важно проговорить это, чтобы у аудитории не было ложного ощущения, будто HTTP «состоит только из CRUD».

---

## Безопасность и идемпотентность

Два важных свойства операций:

* Безопасность (методы не изменяют состояние, например, `GET`, `HEAD`, `OPTIONS`).
* Идемпотентность (повторный вызов метода даёт тот же эффект, например, `PUT`, `DELETE`, безопасные методы).

На практике соблюдение этих свойств позволяет проще организовать повторные запросы.

Note:
Эти свойства критичны для эксплуатационной надёжности API. `Safe` методы позволяют агрессивнее применять кэширование и оптимизации чтения. `Idempotent` методы дают более предсказуемое поведение при сетевых повторах, таймаутах и at-least-once сценариях доставки на уровне инфраструктуры. Проектируя API, полезно заранее думать о повторных вызовах, потому что в продакшене они неизбежны.

---

## Идемпотентность `POST` и `PATCH`

Проблема:

* `POST` и `PATCH` по умолчанию неидемпотентны;
* Повтор запроса может создать дубли (например, заказ/платёж).

Решение:

* Клиент отправляет ключ идемпотентности в заголовке `Idempotency-Key`;
* Сервер сохраняет ключ и результат операции;
* Повтор с тем же ключом возвращает тот же результат, а не создаёт новую сущность.

Note:
Этот паттерн особенно важен в платежных и заказных сценариях, где повторный клик или сетевой retry могут стоить денег. На уровне реализации обычно хранится связка `idempotency_key -> fingerprint запроса -> результат`. Если приходит тот же ключ с другим телом, сервер должен вернуть ошибку конфликта контракта, чтобы не смешивать разные операции под одним идентификатором. Это практический способ сделать опасные `POST`-операции эксплуатационно безопаснее.

---

## Коды ответа HTTP

Часто используемые:

* `200 OK`, `201 Created`, `204 No Content`;
* `304 Not Modified`;
* `400 Bad Request`, `401 Unauthorized`, `403 Forbidden`, `404 Not Found`;
* `409 Conflict`, `422 Unprocessable Content`;
* `500 Internal Server Error`.

Note:
Статус-код — это часть контракта API, а не техническая мелочь. Один и тот же кейс должен возвращать одинаковый класс кода во всех сервисах системы, иначе клиентам приходится писать лишнюю и хрупкую логику. Особое внимание стоит уделять 4xx-кодам и формату тела ошибки: именно там чаще всего возникают недопонимания между backend и потребителями API. Чем последовательнее вы здесь, тем меньше стоимость поддержки интеграций.

---

## Единый формат ошибок API

Для описания ошибок рекомендуется использовать `application/problem+json` ([RFC 7807](https://datatracker.ietf.org/doc/html/rfc7807)).

Базовые поля:

* `type` — тип ошибки;
* `title` — короткое имя;
* `status` — HTTP-статус;
* `detail` — человекочитаемое объяснение;
* `instance` — путь/идентификатор запроса.

Note:
Единый формат ошибок решает очень практичную задачу: клиентам не нужно писать разную логику парсинга ошибок для каждого endpoint-а. RFC 7807 даёт минимальный стандарт, который легко расширяется полями вроде `errors` для валидации и `traceId` для поддержки. Когда команда фиксирует единый формат, интеграции становятся стабильнее, а саппорт быстрее находит причину проблем по логам и трассировке.

---

## Пример `problem+json`

```json []
{
  "type": "https://api.example.com/problems/validation-error",
  "title": "Validation failed",
  "status": 422,
  "detail": "Request body contains invalid fields",
  "instance": "/orders",
  "errors": [
    { "field": "email", "message": "Invalid email format" }
  ]
}
```

Note:
На этом примере важно подчеркнуть баланс между стандартом и расширением. Поля `type/title/status/detail/instance` дают общую основу, а `errors` покрывает прикладную детализацию валидации. Такой контракт хорошо масштабируется: фронтенд может обрабатывать общий каркас единообразно, а специфичные поля использовать там, где это нужно. Главное правило — не ломать базовую форму ответа между сервисами.

---

## Представление ресурса и Content Negotiation

Заголовки:

* `Content-Type` — формат тела запроса/ответа;
* `Accept` — предпочтительный формат ответа.

Примеры:

<div style="display: grid; grid-template-columns: auto auto"><div>

```http []
GET /users/u-42 HTTP/1.1
Host: api.example.com
Accept: application/json
```

</div><div>

```http []
POST /users HTTP/1.1
Host: api.example.com
Content-Type: text/plain

name=Alice
```

</div></div>

Типичные исходы согласования:

* `200 OK` + `Content-Type: application/json` — успешное согласование формата;
* `415 Unsupported Media Type` — сервер не поддерживает формат запроса;
* `406 Not Acceptable` — сервер не может вернуть ответ в формате из `Accept`.

Note:
`Content-Type` и `Accept` отвечают за явную договорённость о формате данных. Для большинства современных API достаточно JSON, но даже в этом случае полезно чётко и последовательно применять эти заголовки в каждом endpoint-е. Важно также различать причины ошибок: `415` означает, что сервер не понимает формат входа, а `406` — что сервер не может отдать ответ в запрошенном клиентом формате. Такая точность делает интеграцию и отладку заметно проще.

---

## Кэширование в REST

Cacheable — одно из ограничений REST и важная часть HTTP-семантики. На уровне API это обычно выражается заголовками:

* `Cache-Control`;
* `ETag`;
* `Last-Modified`.

Note:
На этом слайде фиксируем только связь REST и кэшируемости как архитектурного ограничения. Детали про `Expires`, условные запросы, валидацию через `ETag` и стратегии инвалидирования вынесены в отдельную лекцию, чтобы не перегружать текущую тему.

---

## Layered System и Code-on-Demand

Layered System:

* Клиент не обязан знать, общается ли он напрямую с origin-сервисом;
* Прокси, gateway, cache-серверы естественно встраиваются в архитектуру.

Code-on-Demand (опционально):

* Сервер может передавать исполняемый код клиенту;
* Для REST API в backend-сценариях используется редко.

Note:
Слоистость архитектуры важна для масштаба: вы можете добавлять API gateway, WAF, CDN, кэширующие прокси, не ломая клиентский контракт. Клиент должен видеть единый интерфейс независимо от внутренней топологии. `Code-on-Demand` исторически относится к REST, но в серверных API-практиках почти не используется, поэтому его обычно рассматривают как теоретическую опцию, а не практическое требование.

---

## Параметры запроса

Типовые задачи:

* Фильтрация: `?status=active`;
* Сортировка: `?sort=-createdAt`;
* Пагинация: `?limit=20&offset=40`.

Note:
Работа с коллекциями почти всегда требует фильтров, сортировки и пагинации. Здесь главное — заранее определить единый стиль и не менять его от ресурса к ресурсу без необходимости. Offset-пагинация проста и удобна, но может давать проблемы на больших данных; cursor-подход сложнее, зато стабильнее и эффективнее при высоких объёмах. Ключевой момент для интеграторов — предсказуемость и качественная документация выбранной схемы.

---

## Фильтрация и сортировка

Чтобы не превратить endpoint в SQL-конструктор:

* Разрешать только whitelist полей для фильтрации/сортировки;
* Ограничивать число фильтров и размер страницы;
* Явно фиксировать допустимые операторы (`eq`, `lt`, `in`, ...);
* Валидировать и нормализовать входные параметры.

Основная цель: защита API от "тяжёлых" запросов лимитами и таймаутами.

Note:
Гибкая фильтрация удобна для клиента, но может быстро стать источником проблем производительности и безопасности. Если не ограничить поля и операторы, можно случайно открыть путь к очень дорогим запросам и ухудшить стабильность сервиса. Поэтому фильтрацию лучше проектировать как контролируемый DSL: с чёткой схемой, документированными ограничениями и защитой от злоупотреблений. Это снижает нагрузку на БД и делает поведение API предсказуемым.

---

## Пагинация

Пагинация на сдвигах (`limit` и `offset`):

* Проста для реализации;
* Удобна для случайного доступа к странице;
* Плохо масштабируется на больших смещениях.

Пагинация на курсорах:

* Стабильнее на изменяющихся данных;
* Быстрее на больших объёмах;
* Требует более сложного клиентского контракта.

Важно делать стратегию пагинации стабильной и документированной.

Note:
Offset-пагинация хороша как стартовый вариант, но с ростом данных и конкурентных обновлений начинает давать «скачущие» страницы и деградацию производительности. Cursor-подход лучше подходит для лент и событийных списков, где важны стабильный порядок и эффективность. Для публичного API полезно заранее выбрать один подход по умолчанию и последовательно применять его в похожих endpoint-ах, чтобы не усложнять жизнь клиентам.

---

## Пример курсорной пагинации

```json []
{
  "items": [
    { "id": "o-101", "createdAt": "2026-03-10T12:00:00Z" },
    { "id": "o-102", "createdAt": "2026-03-10T11:58:00Z" }
  ],
  "nextCursor": "eyJjcmVhdGVkQXQiOiIyMDI2LTAzLTEwVDExOjU4OjAwWiIsImlkIjoiby0xMDIifQ==",
  "hasMore": true
}
```

Note:
Этот формат удобен тем, что клиенту не нужно знать внутреннюю механику сортировки, он просто передаёт `nextCursor` в следующий запрос. Важно, чтобы курсор был непрозрачным для клиента и валидировался сервером. Обычно курсор кодирует ключ сортировки и tie-breaker, чтобы порядок оставался стабильным даже при одинаковых значениях основного поля. Такой подход уменьшает риск дублей и пропусков между страницами.

---

## REST и DDD

REST и DDD не противоречат:

* REST — удобный контракт Application Layer;
* DDD — модель и правила Domain Layer.

Application service:

* Принимает HTTP-команду;
* Транслирует её в доменные операции;
* Возвращает HTTP-репрезентацию результата.

Note:
REST отвечает на вопрос «как мы общаемся снаружи», а DDD — «как устроены правила внутри домена». Эти подходы хорошо сочетаются, если слои разделены явно. Контроллер и DTO не должны содержать бизнес-правила, а доменная модель не должна зависеть от особенностей HTTP. В таком разделении проще менять транспортный контракт без разрушения доменной логики и наоборот.

---

## Жизненный цикл и эксплуатация API

<!-- .slide: style="text-align: center" -->

Note:
Финальный блок посвящён эксплуатации API во времени. Разберём безопасность на уровне API, правила совместимости изменений, deprecation-процесс, версионирование, документирование и автоматические контрактные проверки перед релизом.

---

## Безопасность REST API: базовый минимум

Частые риски:

* `BOLA/IDOR` — доступ к чужим объектам по ID;
* Mass assignment / over-posting;
* Недостаточная валидация входных данных;
* Отсутствие rate limiting.

Практика защиты:

* Авторизация на уровне объекта, а не только роли;
* Явные DTO и whitelist полей;
* Лимиты запросов и защитные квоты;
* Аудит чувствительных операций.

Note:
Безопасность API почти всегда ломается не в криптографии, а в прикладной логике доступа и валидации. Самая частая ошибка — проверить, что пользователь авторизован, но не проверить, имеет ли он право на конкретный объект. Вторая типичная ошибка — принимать «как есть» все поля из запроса и непреднамеренно давать клиенту управлять запрещёнными атрибутами. Поэтому в REST-практике важно совмещать строгие DTO, объектную авторизацию и лимиты на злоупотребления.

---

## Версионирование API: когда нужно

Версионирование обязательно, если есть внешние клиенты и риск обратно несовместимых изменений.

Частая практика:

* Достаточно мажорных версий (`v1`, `v2`);
* Минорные версии применяются реже, но тоже возможны.

Note:
Версионирование нужно рассматривать как управленческий инструмент совместимости, а не как формальность. Если у API есть внешние потребители, нельзя рассчитывать на мгновенное обновление всех клиентов при каждом изменении. Поэтому стратегия версий должна быть определена заранее: какие изменения считаются breaking, как долго поддерживаются старые версии, и как команда коммуницирует план вывода версий из эксплуатации.

---

## Совместимость изменений API

Обратно совместимые изменения:

* Добавили необязательное поле в ответ;
* Добавили новый эндпоинт;
* Добавили новое значение перечисления (если клиент готов к `unknown`).

Обратно несовместимые изменения:

* Удалили/переименовали поле;
* Изменили тип или семантику поля;
* Изменили поведение эндпоинта так, что старый клиент ломается.

Note:
Этот слайд помогает команде синхронизировать ожидания: не любое изменение требует новой major-версии, но не каждое «маленькое» изменение безопасно. Критерий простой: если существующий клиент может начать работать некорректно без обновления, это breaking change. Полезно формализовать эти правила в API governance, чтобы решения по версиям принимались последовательно, а не ситуативно.

---

## Версионирование API: как указывать

Основные подходы:

* В URI: `/v1/users`;
* В заголовке: `X-API-Version: 1`;
* Через media type в `Accept`.

Практически самый простой и распространённый — версия в URI.

Note:
У каждого подхода есть свои компромиссы. Версия в URI максимально очевидна и проста для диагностики, логов и документации. Версия в заголовках или media type формально ближе к идеям content negotiation, но часто усложняет поддержку, наблюдаемость и интеграции. В большинстве команд выигрывает самый прозрачный и операционно удобный вариант — явная версия в пути.

---

## Устаревание API (Deprecation)

Версия — это только половина задачи. Нужна ещё политика устаревания:

* Когда эндпоинт считается устаревшим;
* Когда он будет отключён;
* Какой путь миграции предлагается клиенту.

Полезные практики:

* Явное уведомление в changelog и документации;
* Заголовки `Deprecation` и `Sunset`;
* `Link` на актуальную версию/гайд миграции.

Note:
Без формальной deprecation-политики версионирование превращается в хаос: старые версии живут бесконечно, а клиенты узнают об изменениях постфактум. Правильный процесс обычно включает три этапа: объявление устаревания, переходный период, затем отключение. Чем точнее зафиксированы даты, каналы коммуникации и миграционный путь, тем меньше производственных инцидентов при эволюции API. Этот блок особенно важен для публичных или партнёрских интерфейсов.

---

## OpenAPI и Swagger

`OpenAPI`:

* Формальная спецификация API;
* Основа для генерации клиентов, серверных шаблонов и тестовых сценариев.

`Swagger`:

* Набор инструментов вокруг OpenAPI;
* Удобный интерактивный UI для документации и ручного тестирования.

Note:
Спецификация API должна быть не только для людей, но и для инструментов. OpenAPI позволяет превратить документацию в исполняемый артефакт: генерировать клиентские SDK, валидировать контракты и автоматизировать часть тестов. Swagger UI удобен как фронтовая точка входа для разработчиков и интеграторов, но важно поддерживать спецификацию в актуальном состоянии, иначе документация быстро превращается в источник ошибок.

---

## Контрактные тесты API

Зачем:

* Проверять, что provider не ломает ожидания consumer;
* Ловить несовместимые изменения до релиза.

Подходы:

* Consumer-Driven Contracts (например, Pact);
* Проверка OpenAPI-контракта в CI;
* Отдельный compatibility stage перед продом.

Note:
OpenAPI-документация сама по себе не гарантирует совместимость, если её не проверять автоматически. Контрактные тесты закрывают именно этот разрыв: они фиксируют ожидания потребителей и проверяют их против актуального provider-а в пайплайне. Это особенно ценно в экосистеме многих сервисов и внешних клиентов, где ручная проверка всех интеграций невозможна. На практике даже базовый набор контрактных тестов заметно снижает риск регрессий.

---

## Практический разбор эндпоинта

Пример: `POST /orders`

Что должно быть:

* Валидация входа и белый список полей;
* `Idempotency-Key` для защиты от дублей;
* Единый формат ошибок (`problem+json`);
* Авторизация на объектном уровне;
* Версия API и политика устаревания;
* OpenAPI-описание + контрактные тесты.

Note:
Это итоговый интеграционный слайд: он связывает все отдельные практики в единый процесс разработки endpoint-а. Идея в том, что production-ready API — это не только корректный handler и статус-код 200. Это комбинация стабильного контракта, безопасного поведения, предсказуемых ошибок и проверяемой совместимости изменений. Такой шаблон можно использовать как чек-лист при review любого нового endpoint-а в команде.
