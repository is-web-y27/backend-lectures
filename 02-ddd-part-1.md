## Предметно-ориентированное проектирование (DDD)

**Предметно-ориентированное проектирование (Domain-driven design, DDD)**&nbsp;— это подход к разработке ПО, который во главу угла ставит предметную область в целом или какую-то её часть в виде бизнес-процессов.

Note:
Когда программист пишет код без использования DDD, он делает акцент на том, как выполнить ту или иную техническую задачу: отправить что-то в очередь, записать данные в БД и так далее.

Когда появляется DDD, программист начинает смотреть на код через призму бизнеса, т.е. пытается думать так, как бизнес.

---

## Предметно-ориентированное проектирование (DDD)

Какие плюсы даёт такой подход:

* Появляется общая модель всего приложения;
* Бизнес начинает общаться с разработчиками;
* Лучшее понимание UI за счёт понимания потребностей пользователей;
* Лучшее понимание программной архитектуры приложения.

Note:
Плюсов достаточно много, если в DDD вложить время и усилия команды.

## Домен и поддомены

Основной концепцией DDD является домен. **Домен**&nbsp;— это в целом вся предметная область, которая нас интересует.

Для упрощения понимания домен делится на части, которые называются **поддоменами**.

Note:
Основной концепцией DDD является предметная область, её для краткости называют **доменом**.

---

## Типы поддоменов

**Смысловое ядро**&nbsp;— это поддомен, который является ключевым для бизнеса. Для небольших и иногда даже средних приложений оно одно, для крупных приложений их больше.

**Служебный поддомен** также описывает часть бизнеса, но при этом не является настолько важным.
<!-- .element: class="fragment" -->

**Неспециализированный поддомен** не решает задачу именно бизнеса, но при этом всё равно нужен для корректного функционирования приложения.
<!-- .element: class="fragment" -->

Всегда стоит помнить, что поддомен попадает в ту или иную категорию в зависимости от задач, решаемых разрабатываемым продуктом.
<!-- .element: class="fragment" -->

## Единый язык (ubiquitous language)

После выделения поддоменов нужно договориться про их описание, причём в идеале должна быть единая терминология между бизнесом и разработкой, а также в обсуждениях и в коде.

Результатом удачного описания терминологии является единый словарь терминов a.k.a. **единый язык (ubiquitous language)**.
<!-- .element: class="fragment" -->

UL формируется с привлечением доменного эксперта. В идеале UL должен позволить описать термин предметной области без двусмысленности.
<!-- .element: class="fragment" -->

Note:
Выделить поддомены&nbsp;— половина истории. Вторая половина заключается в описании поддоменов. Обычно для этого привлекаются специалисты из предметной области, и вместе с разработчиками они анализируют домен целиком и поддомены. Если вы хорошо слушали курс по анализу и проектированию на UML, то в целом имеете представление, как это работает :)

Если процесс описания идёт хорошо, то на выходе получится **единый язык (ubiquitous language)**. Эта концепция позволяет общаться всем причастным на одном языке.

Ещё раз подчеркнём: для DDD жизненно необходимо активное участие (а не только присутствие) эксперта, разбирающегося в предметной области, для которой разрабатывается ПО. Это либо сам клиент, либо его аналитик.

Формировать единый язык можно разными путями. Обычно начинают со словаря предметной области и продолжают диаграммами (необязательно UML). Сами поддомены тоже являются частью единого языка, и для разных поддоменов может возникнуть необходимость в различающихся определениях и понятиях.

Простой способ проверить, что всё получилось правильно: попробуйте объяснить какой-нибудь термин, используя сформированный язык. В зависимости от того, получится это или нет, можно сказать, правильно ли сформирован UL.

## Ограниченные контексты (Bounded contexts)

В одном поддомене могут возникать разные трактовки терминов.

**Ограниченный контекст (Bounded Context)** фиксирует границы, внутри которых модель и язык согласованы.

<div class="fragment">

Основные задачи ограниченных контекстов:

* изолировать часть модели;
* снизить связность между модулями;
* упростить эволюцию системы.

Поддомены описывают бизнес, контексты описывают реализацию.

</div>

Note:
В целом, когда мы говорим про домен и поддомены, мы всегда отталкиваемся от того, **какие** именно задачи хочет бизнес.

С другой стороны, даже внутри одного поддомена существует ненулевая вероятность разногласий в плане единого языка, и такую ситуацию тоже нужно спроектировать.

Что можно сделать: выделить в рамках поддомена области, в которых единый язык не будет меняться. Такие области называют **ограниченными контекстами (bounded contexts)**.

Основное назначение ограниченных контекстов&nbsp;— инкапсулировать фрагмент предметной области (и его программную реализацию), чтобы минимизировать влияние различных модулей друг на друга. Ещё одна полезная особенность: на один контекст обычно выделяется одна команда разработчиков.

Важно понимать, что поддомены показывают нам структуру предметной области, а ограниченные контексты отражают реализацию. Что стоит помнить при выделении ограниченных контекстов:

* Нет какого-то единственного верного способа выделения;
* Ограниченные контексты могут не соотноситься с поддоменами как один-к-одному.

---

## Ограниченные контексты (Bounded contexts)

Самый понятный пример реализации ограниченных контекстов&nbsp;— микросервисы.

При этом ограниченные контексты не заставляют нас использовать микросервисы, правильный модульный монолит это подтверждает.

Один из наглядных способов выделения ограниченных контекстов: диаграмма использования.

Note:
Самый понятный пример, который можно привести как реализацию ограниченных контекстов&nbsp;— микросервисы. Если немного подумать, то правильно спроектированные микросервисы действительно имеют немного зависимостей и описывают один фрагмент предметной области. Но про них потом :)

При этом микросервисы не являются единственным способом реализации контекстов, главная задача&nbsp;— корректно разделить код всего приложения. То есть правильно спроектированный модульный монолит ничем не хуже.

Для более удобного выделения контекстов можно, например, воспользоваться диаграммой использования и на ней собрать различные варианты использования в подсистемы, эти подсистемы с большой вероятностью будут отвечать за контексты.

## Карта контекстов

Для описания общей интеграции поддоменов существует **карта контекстов (Context Map)**. Основные вопросы, на которые она отвечает:

* где границы контекстов;
* как они взаимодействуют;
* как согласуются термины;
* как распространяются изменения.

Note:
Чем сложнее домен, тем сложнее выделить ограниченные контексты и **связать их**. Редко контекст бывает полностью оторванным
от остальной предметной области, что приводит к нескольким вопросам:

* Где в итоге границы контекстов?
* Как контексты будут взаимодействовать друг с другом?
* Как можно соотнести UL разных контекстов?
* Как изменения одних контекстов затронут другие и как это правильно обрабатывать?

Для этого в DDD есть специальная **карта контекстов**. Это, по сути, верхнеуровневая диаграмма, отвечающая на вопросы выше.

## Карта контекстов и их отношения

Контексты могут взаимодействовать, используя разные шаблоны:

* Партнёрство (Partnership), Общее ядро (Shared Kernel), Заказчик-поставщик (Customer-Supplier);
<!-- .element: class="fragment" -->
* Конформист (Conformist), Предохранительный уровень (Anti-Corruption Layer);
<!-- .element: class="fragment" -->
* Служба с открытым протоколом (Open Host Service), Общедоступный язык (Published Language);
<!-- .element: class="fragment" -->
* Отдельное существование (Separate Ways), Big Ball of Mud.
<!-- .element: class="fragment" -->

Note:
* **Партнёрство** — две команды координируют изменения и совместно развивают интеграцию;
* **Общее ядро** — небольшой общий модуль, который используют несколько контекстов;
* **Заказчик-поставщик** — upstream-контекст публикует модель/API, downstream формулирует требования;
* **Конформист** — downstream принимает модель upstream без собственной адаптации;
* **Предохранительный уровень** — между контекстами ставится слой преобразования и защиты доменной модели;
* **Служба с открытым протоколом** — поставщик публикует стабильный публичный интерфейс интеграции;
* **Общедоступный язык** — контексты договариваются о явном контракте терминов для интеграции;
* **Отдельное существование** — контексты не интегрируются и развиваются независимо;
* **Big ball of mud** — хаотичная взаимозависимая система без чётких границ контекстов.

## Разные объекты в рамках DDD

Когда мы говорим про предметно-ориентированное проектирование, речь обычно заходит о вполне конкретных типах объектов:

* Сущности (entities);
* Значения (value objects);
* Агрегаты (aggregates).

Давайте посмотрим на их отличия.

---

## Сущности

**Сущность (entity)** в рамках DDD&nbsp;— объект, в котором важен его **идентификатор**. Этот идентификатор является неотъемлемым, неизменяемым и уникальным в течение всего времени существования сущности.

Сущность имеет некоторый жизненный цикл и может изменять своё состояние. Как следствие, сущность должна выражать в том числе некоторое **поведение**, а не только данные. Пример плохой сущности: модель из геттеров/сеттеров.

Note:
Что отличает сущность от всего остального:

* Имеет смысл для конечного пользователя и бизнеса в целом;
* Задаётся идентификатором:
  * Две сущности с одинаковым идентификатором считаются одинаковыми, даже если у них отличаются другие поля;
  * Две сущности, отличающиеся только идентификатором, считаются разными;
* Изменяемая с точки зрения хранения в памяти (читай как мутабельная);
* Где-то сохраняется;
* Имеет жизненный цикл.

Когда мы говорим про сущности, мы пытаемся смоделировать не только данные, которые они будут хранить, но и поведение, изменяющее эти данные.

Что имеется в виду: не должно быть ситуации, когда всё изменение модели происходит через пачку геттеров и сеттеров для каждого поля. Нужно описывать те ситуации, которые приводят к изменению данных.

---

```ts []

    public getName(): string {}

    public getDescription(): string {}
    public setDescription(newDescription: string) {}

    public getStatus(): number {}

}
```

Пример плохого кода. Что не так?

Тут ровно то, от чего предостерегает DDD: пачка геттеров и сеттеров. Исправим это...

## Объекты-значения

**Объект-значение (value object)** описывается набором атрибутов, значения которых представляют ценность для бизнеса.

Кроме того, такой объект обычно неизменяем, часто имеет простую структуру и редко используется в отрыве от сущностей.

Note:
В противовес сущностям ставят объекты-значения. **Объект-значение (value object)** описывается не идентификатором (которого нет), а набором атрибутов, значения которых представляют ценность для бизнеса.

Чем VO от сущности:

* Не является чем-то отдельным и независимым, на самом деле, такое значение принадлежит объекту;
* Поскольку нас интересует значение атрибутов, то такие объекты проще заменять, а не изменять, т.е. они неизменяемые;
* Простые по структуре.

---

```ts []
class SubjectName {
    constructor(readonly russianName: string, readonly englishName?: string) {
        if (!russianName.trim()) {
            throw new Error("Название дисциплины не может быть пустым");
        }

        if (englishName !== undefined && !englishName.trim()) {
            throw new Error("Английское название не может быть пустым");
        }
    }
}
  
class SubjectDescription {
    constructor(
      readonly abstract: string,
      readonly topics: string,
      readonly prerequisites: string,
      readonly results: string
    ) {
      if (!abstract.trim() || !topics.trim() || !prerequisites.trim() || !results.trim()) {
        throw new Error("Все части описания дисциплины должны быть заполнены");
      }
    }
}
  
class RejectionReason {
    constructor(readonly reason: string) {
        if (reason.trim().length < 20) {
            throw new Error("Причина отклонения должна быть содержательной");
        }
    }
}
  
class Subject {
    public rename(newName: SubjectName) {}
  
    public setDescription(newDescription: SubjectDescription) {}
  
    public approve() {}
    public reject(reason: RejectionReason) {}
    public archive() {}
}                      
```

## Объекты-значения

Почему это полезно:

* тип сам объясняет смысл поля;
* инварианты находятся рядом с данными;
* код становится безопаснее и понятнее.

Часто даётся совет сначала выделить такие объекты и потом из них собирать сущности.

Note:
Что нам даёт такое выделение значений:

* Тип данных сам за себя говорит, что в нём хранится;
* Можно добавить бизнес-логику, связанную с этим полем, в соответствующий класс;
* Обеспечивается соблюдение инварианта такого объекта-значения.

Часто в литературе даётся совет стараться выделять именно объекты-значения и только потом собирать их в сущности, поскольку это проще разрабатывать и поддерживать.

N.B. Бизнес-логика при этом не должна изменять значение, т.е. все методы должны быть функциями без побочных эффектов.

## Агрегаты

Агрегат (Aggregate) — группа сущностей и/или объектов-значений, которая:

* обрабатывается как единое целое;
* находится в согласованном состоянии;
* идентифицируется по идентификатору корня.

**Корень агрегата**&nbsp;— владелец агрегата, через который идёт взаимодействие с его частями и который обеспечивает согласованность.

Note:
Последнее, что касается проектирования данных&nbsp;— агрегаты. Под **агрегатом** понимается набор сущностей
и объектов-значений со следующими характеристиками:

* Агрегат всегда создаётся, хранится и обрабатывается как единое целое;
* Агрегат всегда находится в согласованном состоянии;
* Владельцем агрегата является сущность, которая называется **корнем агрегата**, её идентификатор
используется для идентификации всего агрегата.

Другими словами, агрегат&nbsp;— это составная сущность, для которой важна транзакционность. Как раз необходимость
транзакционности и определяет то, что мы выделяем в агрегаты, а что оставляем в виде сущностей.

Есть некоторые ограничения, которые накладываются на агрегаты:

* На агрегат можно сослаться только через его корень. Не должно быть ситуации, когда осуществляется доступ к частям агрегата вне самого агрегата.
* За обеспечение согласованности также отвечает корень агрегата.

При проектировании сущностей эти ограничения заставляют думать, какой должна быть сущность: корнем агрегата или частью агрегата. При этом не всегда все сущности обязаны относиться к агрегатам.

## Агрегаты

Практические правила, касающиеся агрегатов:

* Агрегаты должны быть как можно меньше;
* Ссылаться на агрегаты нужно по их идентификатору;
* За одну транзакцию, как правило, изменяется один агрегат;
* При сохранении агрегата нужно помнить про конкурентный доступ.

Note:
Исходя из всего сказанного до этого, можно вывести некоторые правила проектирования агрегатов и работы с ними.

```ts []
class LessonId {
    constructor(public readonly id: string) {
        if (!id.trim()) {
            throw new Error("Идентификатор занятия не может быть пустым");
        }
    }
}
  
class LessonLoad {
      constructor(public readonly load: number) {
          if (load < 0) {
              throw new Error("Нагрузка не должна быть отрицательной");
          }
      }
}
  
class LessonEvent {
    constructor(private dateStart: Date, private dateEnd: Date) {
        if (dateStart >= dateEnd) {
            throw new Error("Начало занятия должно быть раньше окончания");
        }
    }

    public intersects(other: LessonEvent): boolean {
        return this.dateStart < other.dateEnd && other.dateStart < this.dateEnd;
    }
}
  
class Lesson {
    constructor(
      private id: LessonId,
      private slots: LessonEvent[] = [],
      private totalLoad: LessonLoad = new LessonLoad(0)
    ) {}
  
    public addEvent(dateStart: Date, dateEnd: Date) {
      const newEvent = new LessonEvent(dateStart, dateEnd);

      if (this.slots.some(slot => slot.intersects(newEvent))) {
          throw new Error("События занятия не должны пересекаться");
      }

      this.slots.push(newEvent);
  
      // Каждая пара считается за 2 единицы нагрузки
      this.totalLoad = new LessonLoad(this.totalLoad.load + 2);
    }
}
  
```
